HTTP协议的核心特点这个东西说实话已经被讨论过太多次，然而大多数都是基于八股文那般照本宣科，未曾思考其中蕴意。在经过诸多实践和思考，我决定分享一下我对Http协议的核心特点的更多思考
## 普遍特性
1. 请求-响应模式，预期说是特点，不如说是缺点。所谓请求-响应模式，即通信只能由客户端发起 ，例如，我们想了解今天的天气，只能是客户端向服务器发出请求，服务器返回查询结果。这里得区分keep-alive字段下的长连接。须注意到长连接只是服用TCP连接，但是通信模式依旧是机械刻板的请求响应，一问一答。也正是因为这个模式，在应用层协议书诞生了一个Websocket协议，该协议说白了就是，你说你的，我说我的。
在WebSocket中，先通过 HTTP 发起 Upgrade，注意初期还是HTTP协议，升级成功后，这条连接变成 WebSocket 连接，之后双方可以持续互发消息。


2. 无状态（Stateless）HTTP协议本身不保留任何历史请求的信息。每个请求都是独立的，服务器在处理请求时无法直接区分多个请求是否来自同一客户端。这种设计简化了服务器实现，但也带来了状态管理的需求，不过好在由于其灵活可扩展的特性，使得在这个协议上无状态的协议能够实现有状态需求的业务。

3. 传输层协议（TCP/UDP）实现可靠或低延迟的数据传输，本身只关注数据的内容和语义。

4. 灵活可扩展，通过头部字段（Header）可以携带元数据，支持自定义扩展，如Cookie、缓存指令、内容协商等。HTTP方法（GET、POST、PUT等）和状态码的设计也为扩展提供了空间。

这些特性乃是Http的通用特性，Http协议实际上经历多次迭代，不同版本之间亦有差别
## 版本特性
1. HTTP/1.0（1996）
    短连接：每个HTTP请求/响应都需建立新的TCP连接，完成后立即断开。频繁的TCP握手和慢启动导致性能低下。

    文本格式：头部和体均为纯文本，解析效率较低。

    无主机头（Host）：无法在一台服务器上托管多个域名，直到后来通过
    Host
    头扩展。

2. HTTP/1.1（1999）
持久连接（Keep-Alive）：默认启用，允许多个请求复用同一个TCP连接，显著减少握手开销。

    管道化（Pipelining）：理论上客户端可在同一连接上连续发送多个请求而不等待响应，但队头阻塞（HOL blocking）问题——因为响应必须按请求顺序返回，一旦前面的请求处理慢，后续响应都会被阻塞，注意这点和Http本身是无状态协议嘻嘻相关。

    增强的缓存控制：引入
    Cache-Contro
    ETag
    等头部，缓存机制更精细。

    分块传输编码（Chunked）：允许动态生成内容时边生成边发送。

    强制Host头：支持虚拟主机托管。

3. HTTP/2（2015）
    最牛的——二进制分帧：传输单元改为二进制帧（如HEADERS帧、DATA帧），取代文本格式，解析更高效。因为HTTP/1.1 文本格式需要不断查找空格、换行、冒号等分隔符，解析规则更复杂。二进制帧的结构固定，机器可以直接寻找第几字节到第几字节长度直接读取；

    在分帧的基础上实现多路复用：在单一TCP连接上同时交错发送多个请求/响应，每个请求被拆分为带流ID的帧，接收方重组。彻底解决了应用层的队头阻塞。

    头部压缩（HPACK）：使用静态表、动态表和Huffman编码压缩头部，减少冗余传输（尤其适用于大量小资源请求）。

    服务器推送：服务器可预测客户端需要哪些资源（如HTML中引用的CSS、JS），主动推送，减少往返延迟。

    流优先级和依赖：允许客户端指定资源加载优先级。
